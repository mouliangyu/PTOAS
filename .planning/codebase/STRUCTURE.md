# Codebase Structure

**Analysis Date:** 2026-03-18

## Directory Layout

```text
PTOAS/
├── include/              # Public C++ headers and TableGen definitions for PTO
├── lib/                  # Dialect, pass, C API, and Python-binding implementations
├── tools/                # CLI frontends and PTOBC support library
├── python/               # Python packaging files copied into the MLIR Python layout
├── test/                 # MLIR tests, sample programs, compile checks, and NPU validation assets
├── cmake/                # Install/export package config templates
├── docs/                 # Project documentation
├── docker/               # Container-related setup
├── build/                # Local build tree generated by CMake
├── install/              # Local install tree generated by `cmake --install`
├── openspec/             # Planning/spec artifacts for changes
└── .planning/codebase/   # Generated codebase mapping documents
```

## Directory Purposes

**`include/`:**
- Purpose: Keep public compiler interfaces and TableGen inputs together.
- Contains: `include/PTO/IR/` dialect headers and `.td` files, `include/PTO/Transforms/` pass declarations, and `include/pto-c/` C API headers.
- Key files: `include/PTO/IR/PTO.h`, `include/PTO/IR/PTOOps.td`, `include/PTO/Transforms/Passes.h`, `include/pto-c/Dialect/PTO.h`

**`lib/`:**
- Purpose: Implement the interfaces declared in `include/`.
- Contains: `lib/PTO/IR/` dialect code, `lib/PTO/Transforms/` passes, `lib/CAPI/` C API implementation, and `lib/Bindings/Python/` pybind11 bindings.
- Key files: `lib/PTO/IR/PTO.cpp`, `lib/PTO/Transforms/PTOToEmitC.cpp`, `lib/PTO/Transforms/PTOPlanMemory.cpp`, `lib/CAPI/Dialect/PTO.cpp`, `lib/Bindings/Python/PTOModule.cpp`

**`tools/`:**
- Purpose: Hold executable frontends and tool-specific support code.
- Contains: `tools/ptoas/` compiler CLI, `tools/ptobc/` encoder/decoder CLI plus shared bytecode helpers and tests.
- Key files: `tools/ptoas/ptoas.cpp`, `tools/ptobc/src/main.cpp`, `tools/ptobc/src/mlir_encode.cpp`, `tools/ptobc/tests/CMakeLists.txt`

**`python/`:**
- Purpose: Provide handwritten Python package files layered on top of generated MLIR Python bindings.
- Contains: CMake glue and the `python/pto/dialects/pto.py` module copied into the built package.
- Key files: `python/CMakeLists.txt`, `python/pto/dialects/pto.py`, `python/pto/dialects/PTOOps.td`

**`test/`:**
- Purpose: Group verification material by test style rather than by production layer.
- Contains: `test/basic/` focused `.mlir` inputs, `test/samples/` many sample kernels and generated outputs, `test/compile_cpp/` compile checks for emitted C++, and `test/npu_validation/` templated remote validation scripts.
- Key files: `test/basic/record_wait_event.mlir`, `test/samples/MatMul/tmatmulk.py`, `test/compile_cpp/compile_lib_full_flags.sh`, `test/npu_validation/scripts/generate_testcase.py`

**`cmake/`:**
- Purpose: Store package export templates consumed during install.
- Contains: CMake config templates.
- Key files: `cmake/PTOASConfig.cmake.in`

**`build/`:**
- Purpose: Hold generated build outputs, generated headers, binaries, and temporary test artifacts.
- Contains: generated `include/`, built libraries/tools, and CTest metadata.
- Key files: `build/tools/ptoas/ptoas`, `build/python/mlir/_mlir_libs/`, `build/include/PTO/`

**`install/`:**
- Purpose: Hold locally installed headers, libraries, and Python artifacts.
- Contains: exported CMake config, installed headers, installed libraries, and `mlir/dialects/` Python files.
- Key files: `install/lib64/cmake/PTOAS/`, `install/include/PTO/`, `install/mlir/dialects/`

## Key File Locations

**Entry Points:**
- `CMakeLists.txt`: Root build entry point for the whole toolchain.
- `tools/ptoas/ptoas.cpp`: Main PTO compiler executable.
- `tools/ptobc/src/main.cpp`: PTOBC encode/decode executable.
- `lib/Bindings/Python/PTOModule.cpp`: Python extension entry point.

**Configuration:**
- `CMakeLists.txt`: Global options, dependency discovery, install/export setup.
- `tools/ptoas/CMakeLists.txt`: Build definition for the main compiler binary.
- `tools/ptobc/CMakeLists.txt`: Build definition for the PTOBC support library and binary.
- `python/CMakeLists.txt`: Python target orchestration and generated module handling.

**Core Logic:**
- `include/PTO/IR/` and `lib/PTO/IR/`: PTO dialect contract and implementation.
- `include/PTO/Transforms/` and `lib/PTO/Transforms/`: pass APIs and implementations.
- `lib/PTO/Transforms/InsertSync/`: synchronization analysis/codegen subsystem.
- `tools/ptobc/src/`: bytecode codec implementation.

**Testing:**
- `test/basic/`: small MLIR fixtures for pass and IR behavior.
- `test/samples/`: representative PTO programs in Python, PTO, and emitted C++.
- `tools/ptobc/tests/`: CTest-integrated smoke/e2e checks for PTOBC.
- `test/npu_validation/`: generated-on-demand validation scaffolding.

## Naming Conventions

**Files:**
- Use `PascalCase` for major pass implementation files and dialect headers: `PTOPlanMemory.cpp`, `PTOToEmitC.cpp`, `PTODialect.h`.
- Use feature-group subdirectories when a subsystem is large enough to need multiple helper files: `lib/PTO/Transforms/InsertSync/SyncCodegen.cpp`.
- Use lowercase for tool entrypoints and low-level helper files: `tools/ptoas/ptoas.cpp`, `tools/ptobc/src/mlir_encode.cpp`, `tools/ptobc/src/leb128.cpp`.
- Use MLIR-standard `.td` names for generated IR/pass inputs: `PTOOps.td`, `PTOTypeDefs.td`, `Passes.td`.

**Directories:**
- Mirror public and private layers by name: `include/PTO/IR/` pairs with `lib/PTO/IR/`, and `include/PTO/Transforms/` pairs with `lib/PTO/Transforms/`.
- Keep executable-specific code under `tools/<tool-name>/` instead of mixing it into library directories.
- Keep sample and validation assets under `test/` even when they contain scripts or generated C++.

## Where to Add New Code

**New Feature:**
- Primary code: add new dialect ops/types/attrs in `include/PTO/IR/` and implement them in `lib/PTO/IR/` if the feature changes the PTO language surface.
- Primary code: add new passes in `include/PTO/Transforms/Passes.h`, `include/PTO/Transforms/Passes.td`, and `lib/PTO/Transforms/` if the feature is a transformation or lowering.
- Tests: add focused IR fixtures under `test/basic/` and end-to-end or sample-style coverage under `test/samples/`.

**New Component/Module:**
- Implementation: place reusable compiler libraries under `lib/PTO/<area>/` only when the code is library-grade and is consumed by more than one tool.
- Implementation: place tool-only helpers under the owning tool directory, for example `tools/ptobc/src/` for PTOBC-specific codec logic.

**Utilities:**
- Shared helpers: keep transform-local helpers next to the owning pass in `lib/PTO/Transforms/` or `lib/PTO/Transforms/InsertSync/`.
- Shared helpers: keep dialect-wide helpers next to the dialect in `lib/PTO/IR/` and expose them through `include/PTO/IR/` only if they are part of the public library surface.

## Special Directories

**`include/PTO/IR/`:**
- Purpose: Source-of-truth IR declarations and TableGen inputs.
- Generated: No
- Committed: Yes

**`build/include/PTO/`:**
- Purpose: Generated `.inc` headers emitted by TableGen during the build.
- Generated: Yes
- Committed: No

**`lib/PTO/Transforms/InsertSync/`:**
- Purpose: Multi-file implementation of the auto-sync subsystem.
- Generated: No
- Committed: Yes

**`tools/ptobc/generated/`:**
- Purpose: Checked-in generated opcode definitions used by the PTOBC implementation.
- Generated: Yes
- Committed: Yes

**`test/npu_validation/templates/`:**
- Purpose: Template files used to generate per-sample NPU validation harnesses.
- Generated: No
- Committed: Yes

**`build/`:**
- Purpose: Local compiler outputs and intermediate artifacts.
- Generated: Yes
- Committed: No

**`install/`:**
- Purpose: Local install prefix used for testing packaging and runtime setup.
- Generated: Yes
- Committed: No
